AI 安全 Spec 规范指南

└─────────────────────────────┘

一、安全 Spec 应具备的特征

  1. 范围与对象清楚
    写明适用系统（如内外部 Web、移动端、Agent 场景）、技术栈（Java/Node 等）和边界（不含哪些，如纯物理安全）。避免「万能文档」导致无法执行。

  2. 可落地、可检查
    不只写原则，还要有：必须 / 禁止、错误示例与正确示例、与框架/组件相关的配置要点。便于开发对照实现，也便于审查时问「是否满足哪一条」。

  3. 与威胁与标准对齐
    显式对齐 STRIDE、OWASP、等保或内部基线，让读者知道「这条要求从哪类风险来」。新同事和审计方都能快速建立信任。

  4. 分级与优先级
    区分高危 / 中危 / 低危或「必须修复才能上线 / 可接受风险」等，避免所有条款同等加粗，导致无法排期。

  5. 覆盖完整生命周期
    至少包含：设计阶段（威胁建模）、编码（注入、鉴权、会话、文件、反序列化、SSRF 等）、传输与配置、日志与异常、CI/发布前检查。缺了「上线前怎么验」容易流于纸面。

  6. 新兴攻击面
    对当前组织真实存在的形态要有专章或条款，例如 AI Agent / 提示词注入、工具调用边界，否则规范会落后于实际研发方式。

  7. 可追踪、可演进
    版本号、更新日期、变更记录；条款最好带稳定锚点或 ID，便于 PR 评论、工单和合规报告引用同一句话。

  8. 人机两用（可选但很有价值）
    除长文 Markdown 外，用 结构化数据（如 YAML/规则 ID）描述「规则 ID、严重级别、正反模式、对应规范章节」，供 SAST、CI、内部扫描或智能体消费——这就是你们 security-rules.yaml 一类文件的意义。

二、安全 Spec 发挥的作用

  1. 统一「安全底线」
    跨团队、外包、新同学对「什么算违规、什么必须做」有同一套依据，减少口头传说和各自理解。

  2. 把安全左移到设计与编码
    在设计/评审阶段就能对照条款（如鉴权、SSRF、敏感数据），比在上线前堆渗透测试更省成本。

  3. 支撑代码审查与审计
    审查者可引用具体章节；内外部审计可用同一文档说明控制措施与覆盖范围。

  4. 驱动自动化
    结构化规则可映射到 Sonar、依赖扫描、镜像扫描、流水线门禁，把「规范」变成「合并前必过」的检查，而不是只靠自觉。

  5. 培训与智能体上下文
    新人 on-boarding、安全培训和 AI 辅助编码 都可以把 Spec 当作权威上下文，减少模型「凭感觉」给出不安全建议。

  6. 风险与合规的可沟通载体
    对管理层和业务方，Spec 是「我们承诺做到哪些控制」的简明版本（再辅以分级与例外流程）。

三、为什么要有 Markdown？

主要给人读、给人写、给人讨论。

  • 叙述完整：能写背景、威胁模型、原则、表格、流程和「为什么」，而不只是一条条字段。
  • 示例友好：代码块、反例/正例、配置片段都适合放在 MD 里，开发一眼能对上自己用的框架。
  • 评审与培训：PR 里 diff 一段说明、评审会上打开一章，都比对着 YAML 键值讨论更自然。
  • 版本与法务痕迹：长文规范常常是审计、外包合同里「遵循某某文档」的引用对象。

所以 CODING_SECURITY_SPEC.md 这类文件承担的是「权威叙述 + 教学 + 约定细节」。

四、为什么要有 YAML？

主要给程序读、给流水线读、给工具链读。

  • 结构固定：id、severity、spec_ref、anti_pattern 等字段稳定，脚本、SAST 包装、内部门户都能直接解析。
  • 易集成：CI 里 yq/jq、自定义 linter、看板统计「多少条 HIGH」、按 category 过滤，都依赖结构化数据。
  • 体量小、查询快：不需要解析整篇 Markdown 标题树才能知道「有哪些安全规则」。
  • 多工具同一套清单：Sonar、自研扫描、Agent 策略可以共用一份 YAML，而不是各写各的配置。

所以 meta/security-rules.yaml 承担的是「机器可消费的规则索引 / 检查项清单」。

五、为什么两种都要？

读者和用途不同，一种格式很难同时兼顾。

维度 Markdown YAML
最佳读者 人 程序 / CI / 工具
强项 解释、示例、上下文 字段化、可查询、可自动化
弱项 难稳定解析成「规则列表」 不适合写长文和大量代码示例

只保留 MD：自动化要嘛手写重复配置，要嘛每次从 MD 里「猜」结构，脆弱且易漂移。
只保留 YAML：新人很难理解「为什么」「边界在哪」「具体怎么配 Spring」。

两者组合 = 叙述真源 + 机器快照，是常见做法。

六、它们如何联系？

在你们仓库里，联系主要靠 显式指针和约定，而不是魔法自动生成（除非以后做生成器）。

  1. YAML → MD
    每条规则的 spec_ref（例如 CODING_SECURITY_SPEC.md#…）指向规范里的章节锚点，工具报错时能链回长文说明。

  2. 索引里声明关系
    meta/index.yaml 里 CODING_SECURITY_SPEC 的 meta 会列出 security-rules.yaml，meta_files 里可有 generated_from: CODING_SECURITY_SPEC.md，说明「这条 YAML 在语义上从哪份规范来」。

  3. 维护纪律（最重要）
    README 里要求：规范里若出现结构化内容（安全规则、版本约束等），要同步维护 meta/。
    也就是说：联系是「文档约定 + 人工/流程保持同步」；改 MD 里某条要求时，应检查对应 YAML 是否仍一致。
